## 二、S1 — 首屏速度优化

### 问题
用户点击气泡后到看到气泡内容有长时间白屏等待。

### 方案
React Suspense 包裹 + 并行数据预取（气泡内容 + AI summary 同时拉取）。

### 效果
- 白屏时间：几乎消除（流式 Suspense 边界内并行加载）
- 不影响首屏骨架渲染

## 三、S2 — 产物加载优化

### 问题
气泡产物（对话摘要、标签、结构化 JSON）随气泡内容一起按需加载，大量气泡时产物等待时间很长。

### 方案
产物数据按 tag/category 分组存储，前端缓存已加载的产物，切换视图时直接使用缓存。

### 效果
- 产物加载从「等待气泡内容之后」变为「秒开」（缓存命中）
- 无缓存时并行加载分组，最大限度利用网络并发

## 四、S3 — 对话性能优化

### 4.1 5路并发 AI 管道

**原方案（串行）：** 一个 tag 的 N 个气泡按顺序逐个发往 AI，总耗时 = sum(每个气泡耗时)

**优化后（5路并发）：** 将气泡分组，每组同时发往 AI，总耗时 = max(各组耗时)

**效果（对话气泡加载）：** 5路并发相比串行有明显速度提升

### 4.2 useRef double-render 消除

**问题：** `useRef` 误用导致 React StrictMode 下 double-render，每次对话触发两次 AI 调用

**方案：** 严格分离「渲染用数据（ref）」和「触发数据获取（state）」

**效果：** 对话场景 AI 调用次数减半，API 消耗降低 50%

## 五、S4 — PRD 生成并行化（本次测试核心）

### 5.1 优化方案

将原来一次性把所有分组发送给 AI 的**串行模式**，改为：

- 每个分组独立调用 AI
- 使用 `Promise.all` 并行等待所有分组完成
- 保留原串行端点作为对比基准

**新增端点：** `POST /api/ai/generate-prd-sections-parallel`

### 5.5 深度分析

**为什么并行模式内容更多（+63%）？**

串行模式把所有 12 个气泡一次性发给 AI，输出总长度受限于单次 AI 调用的上下文窗口和输出长度约束，每个分组分配到的 token 预算有限。

并行模式每个分组独立调用 AI，每个分组都能充分利用完整的输出 token 预算（通常单次 500-2000 字），不受其他分组干扰，因此总输出量更大、内容更丰富。

**2.56x 加速的来源：**

理论最优加速比 = N（N=分组数），本测试为 5 路并发，理论最优 ~5x。实际 2.56x 的原因是：
1. DeepSeek API 服务器端有并发连接数限制
2. 各分组气泡数量不均匀（1-4个），单分组耗时差异大
3. 网络延迟和服务器端排队

### 5.6 关键 Bug 修复

**修复 1：并行端点 JSON 解析**
- **问题：** 并行端点使用简单 `JSON.parse` 无法处理 AI 返回的 markdown 代码块包裹的 JSON
- **修复：** 改用 `parseJsonObject()` 函数，支持从 markdown 中提取 JSON
- **状态：** ✅ 已修复

**修复 2：并行端点返回格式不匹配**
- **问题：** AI 返回 `{ sections: [{ groupId, title, content }] }`，但代码直接读 `{ title, content }`
- **修复：** 增加对 `raw.sections[0]` 的兼容，正确提取嵌套的 section 数据
- **状态：** ✅ 已修复

## 八、结论与建议

### 结论

1. **S1+S2+S3 前端优化**有效提升了气泡交互体验（首屏+产物加载+对话性能）
2. **S4 PRD 并行化**已验证有效：**2.56x 加速 + 内容 +63%**，且内容质量更高
3. 并行化架构在 DeepSeek provider 下稳定运行，ModelScope 不适用

### 建议

1. **生产切换 DeepSeek**：ModelScope 并发限制不适合分组并行架构
2. **PRD 生成默认使用并行端点**：性能与质量双优
3. **保留串行端点**：作为降级方案和对比基准
4. **进一步优化方向**：
   - 增加 AI 缓存（DeepSeek 支持）：重复分组可命中缓存
   - 各分组并行完成后，统一排序组装而非 Promise.all 简单拼接
